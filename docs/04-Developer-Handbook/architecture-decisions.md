# Architecture Decisions

## Purpose

Record repository-level engineering decisions that should outlive a single task.

## When to Add a Decision

- a new repository-wide pattern is introduced
- a workflow or merge rule changes
- a persistent tradeoff is accepted
- future contributors would otherwise relearn the same context

## Lightweight ADR Format

- Title
- Status
- Date
- Context
- Decision
- Consequences
- Links

## Placement

- architecture choices affecting the product belong in `docs/01-Architecture/`
- process and governance choices belong in this handbook

## Rule

Do not create an ADR for routine task output. Create one when the decision will
shape future work.
